Debriefing

1. Classe abstraite

Une classe abstraite est par définition une classe dont les attributs et méthodes ne suffisent pas à représenter un objet réel. Dans le problème des formes géométriques, la classe Forme est une classe abstaite, car elle ne possède pas de méthode trace(). Et en effet, il est impossible de coder une telle méthode, qui serait capable de tracer n’importe quelle forme géométrique. Dans le problème de biodiversité, la classe Animal est, pour les mêmes raisons, une classe abstraite : elle ne possède pas de méthode bouge() et crie().

La théorie stipule qu’une classe abstraite n’a pas vocation à être instanciée : sa fonction est de servir de classe de base pour dériver d’autres classes. Or, dans les 2 cas, rien n’empêche le développeur de créer ces instances :

un_gribouillage = Forme(g=(50,50),l=100)   # ok
un_gribouillage.trace()  # AttributeError: 'Forme' object has no attribute 'trace'
une_licorne = Animal()   # ok
une_licorne.crie()       # AttributeError: 'Animal' object has no attribute 'crie'

Conclusion : il est possible de créer des instances de ses classes abstraites, mais celles-ci posent problème assez rapidement.

2. Hériter d’une classe abstraite

Une classe qui hérite d’une classe abstraite n’est plus abstraite si elle implémente tous les manques de sa classe de base. Par exemple, la classe :

class Poisson(Animal):
  def bouge(self):
    return "je nage"
  def crie(self):
    return "..."

n’est plus une classe abstraite. Par contre, une classe qui hérite d’une classe abstraite et qui n’implémente tous ses manques reste abtraite :

class Mammifere(Animal):
  def bouge(self):
    return "je marche"

Mammifere est abtraite, car elle ne possède pas (ni directement, ni par héritage) de méthode crie().

3. Etayer le concept de classe abtraite

Le concepteur de la classe Animal peut tenter de signifier aux développeurs qui l’utiliseront que cette classe est abtraite :

class Animal:
  def crie(self):
    raise NotImplementedError("cette méthode doit être implémentée")
  def bouge(self):
    raise NotImplementedError("cette méthode doit être implémentée")

ce qui donne :

class Licorne(Animal):
    pass

une_licorne = Licorne()   # ok
une_licorne.crie()        # NotImplementedError: cette méthode doit être implémentée

Ainsi, les développeurs en question comprendront qu’il ne s’agit pas d’un bug, et que c’est à eux de faire le boulot. Il est en effett plus clair que l’instruction une_licorne.crie() renvoie "NotImplementedError: cette méthode doit être implémentée" plutôt que "AttributeError: Licorne object has no attribute crie".

Le module standard abc (pour abstract class) permet de faire encore mieux :

from abc import ABC, abstractmethod

class Animal(ABC):
  @abstractmethod
  def bouge(self):
    raise NotImplementedError()

  @abstractmethod
  def crie(self):
    raise NotImplementedError()

Ici, on indique explicitement que Animal est abstraite, ainsi que les 2 methodes bouge et crie, ce qui donne comme nouveau comportement :

a = Animal()      # TypeError: Can't instantiate abstract class Animal with abstract methods bouge, crie
m = Mammifere()   # TypeError: Can't instantiate abstract class Mammifere with abstract method crie

Plus d’ambigüité, c’est plus clair !
